# NewsMon — аудит поточного стану проєкту (квітень 2026)

## 1. Резюме

Проєкт перебуває у стані **робочого прототипу з production-орієнтованими елементами**:
- є реальний ingestion через Telethon,
- є асинхронна AI-черга з Claude,
- є UI для операційного керування,
- впроваджено базову адмін-аутентифікацію через Bearer-токен.

Водночас зберігається сильна концентрація логіки в одному модулі `backend/app.py`, відсутні автотести та не завершені частини ТЗ (alerts/workflow/search/ролі).

## 2. Що реалізовано добре

### 2.1 Бекенд-ядро
- FastAPI API із повним CRUD для джерел, категорій, ключових слів.
- Монітор-цикл (`_monitor_loop`) із періодичним опитуванням каналів.
- Ручний тригер інжесту (`POST /api/monitor/run-once`).
- AI-процесинг через `ai_queue` з claim-механікою `pending -> processing`.

### 2.2 Схема даних
- SQLite WAL + FTS5.
- Нормальний набір індексів для типових читань dashboard/filters.
- Закладені таблиці для майбутніх модулів: alerts, bookmarks, users, media.

### 2.3 Стійкість Telethon
- Глобальний lock для клієнта та обробки AI.
- Валідація/health-check session.
- Quarantine пошкодженої session БД.
- Rate limit на auth-ендпоінтах.

### 2.4 UX оператора
- Dashboard читає лише `ai_status='done'` та має модалку деталей.
- Settings має вкладки для інтеграцій/моніторингу/таксономії/джерел/debug.
- Реалізований фронтовий auth-flow для адмін-токена.

## 3. Ключові ризики

1. **Монолітність `app.py`** — один файл поєднує API, доменну логіку, зовнішні інтеграції, планувальник і security.
2. **Відсутність автотестів** — високий ризик регресій при подальших змінах ingestion та Telethon auth-flow.
3. **Безпека рівня MVP** — один shared-token без ролей/ротації/аудит-логу дій.
4. **Операційний ризик SQLite** на довгій дистанції при рості обсягів і паралельних операціях.
5. **Неповна відповідність ТЗ** — значна частина roadmap (workflow, alerts engine, пошук, аналітика, ролі) ще не імплементована.

## 4. Розрив між ТЗ та фактом

### Уже є
- Telegram ingestion + Telethon auth,
- AI scoring + категоризація,
- monitor config (interval/fetch depth/ai prompt),
- status/debug API,
- прототипний UI dashboard/settings.

### Ще не закрито
- Рольова модель (Admin/Editor/Reader) на бекенді.
- Реальні alerts правила/движок (зараз у UI секція здебільшого mock).
- Розширений пошук (в API немає окремого endpoint поверх FTS5).
- Workflow-операції над повідомленнями ("взяв у роботу", "опрацьовано" як повний життєвий цикл).
- Аналітика/експорт/закладки як завершені модулі.

## 5. Пріоритетний план покращень

### P0 (негайно, до активної експлуатації)
1. Винести модулі з `app.py`:
   - `routers/*`, `services/telethon_service.py`, `services/ai_service.py`, `security.py`.
2. Додати мінімальний test-suite:
   - unit: валідації, парсери, `_prepare_ai_text`,
   - integration: ключові API-ендпоінти,
   - smoke: monitor run-once + queue processing (з моками).
3. Впровадити structured logging + базовий audit log адмін-операцій.

### P1 (короткий спринт)
1. Реалізувати FTS-пошук в API та UI.
2. Добудувати workflow-статуси повідомлень і відповідні ендпоінти.
3. Завершити alerts engine (rule match + канали доставки).

### P2 (стабілізація)
1. Метрики продуктивності та витрат Claude (cost controls, caps).
2. Механізми backup/retention на рівні сервісу.
3. Підготовка до переходу на більш масштабований storage у разі росту навантаження.

## 6. Підсумкова оцінка

Поточна реалізація виглядає як **сильний функціональний прототип (MVP+)** з реальним ingestion та AI-обробкою, який уже придатний для обмеженого внутрішнього використання. Для надійної експлуатації в редакції на постійній основі потрібно закрити P0-блок: модульність, автотести, операційна спостережуваність і посилення security-моделі.
